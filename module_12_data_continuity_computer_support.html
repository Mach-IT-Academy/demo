🎯 Final Challenge: Multi-Scenario

A company runs a full backup every Monday and incremental backups Tuesday–Friday. A disaster occurs Friday afternoon. How many backup sets are needed to fully restore?

Just Friday's backup
Monday's full + Friday's incremental only
Monday's full + Tuesday + Wednesday + Thursday + Friday incrementals
✅ Full chain is correct. With incremental backups, each day's backup only contains what changed since the previous day. To restore Friday's state, you need ALL incrementals in the chain: Mon (full) + Tue + Wed + Thu + Fri. This is the key weakness of incrementals — complex restores.

Management says: "We need to be fully operational again within 2 hours of any disaster." What DR metric does this represent, and what site type likely supports it?

RPO of 2 hours → Cold Site
RTO of 2 hours → Warm or Hot Site
MTBF of 2 hours → any site type
✅ Correct! "Back operational within X time" = RTO. A 2-hour RTO requires either a hot site (fastest, near-instant) or a warm site (equipment ready, data must be loaded). A cold site typically takes days, making it incompatible with a 2-hour RTO.

📋 Scenario

A storage consultant is configuring four servers for different departments. Each department has distinct performance and redundancy requirements. The consultant must match each description to the correct RAID level before the hardware deployment window opens.

Task: Drag each scenario description from the right pool and drop it onto the correct RAID level on the left. Each zone accepts exactly one item.
⚡ RAID 0 — Striping
🪞 RAID 1 — Mirroring
🔐 RAID 5 — Striping + Parity
🏆 RAID 10 — Mirror + Stripe
⠿Animation studio scratch disk — maximum read/write speed required; data is re-rendered from source files if lost; zero redundancy acceptable
⠿Boot drive for a small business file server — must tolerate a single drive failure without any downtime; 2 drives, 50% usable capacity
⠿Financial trading database — requires both maximum fault tolerance AND high-speed random read/write; minimum four drives; willing to sacrifice 50% storage
⠿NAS for a law firm's shared documents — efficient storage use, survives one drive failure, good read performance; minimum three drives required

✅ Why Each RAID Level Is Correct

  • RAID 0 — Animation Scratch Disk: RAID 0 stripes data across all drives simultaneously, delivering the highest possible throughput. There is zero redundancy — if any single drive fails, ALL data is lost. This is acceptable for scratch/temp storage where source files exist elsewhere. RAID 0 uses 100% of available drive capacity.
  • RAID 1 — Small Business Boot Drive: RAID 1 writes identical data to two drives simultaneously (mirroring). If one drive fails, the other continues operating without interruption. The tradeoff is 50% usable capacity (2 × 1 TB = 1 TB usable). Minimum 2 drives. Ideal for OS drives and critical single-server deployments.
  • RAID 5 — Law Firm NAS: RAID 5 stripes data with distributed parity across a minimum of 3 drives. If one drive fails, the parity data on the remaining drives can reconstruct everything. Usable capacity is (n-1) drives. Great read performance; write performance is slightly reduced due to parity calculation. Most widely deployed RAID for general-purpose storage.
  • RAID 10 — Trading Database: RAID 10 (also called RAID 1+0) mirrors pairs of drives and then stripes across those mirrors. This combines RAID 1's fault tolerance with RAID 0's performance. Requires minimum 4 drives; 50% usable. Can survive multiple simultaneous drive failures as long as both drives in a mirrored pair don't fail. Premium option for databases and high-IOPS workloads.

❌ Common Mistakes

  • Treating RAID as a backup: RAID provides fault tolerance against hardware failure — it does NOT protect against accidental deletion, ransomware, or site-wide disasters. You still need backups regardless of RAID level.
  • Confusing RAID 5 and RAID 10 for databases: RAID 5 can handle one failure but has slower write speeds (parity overhead). High-performance databases need RAID 10 for fast random I/O.
  • Minimum drive counts: RAID 0 = 2 drives min · RAID 1 = 2 drives min · RAID 5 = 3 drives min · RAID 10 = 4 drives min.

🌍 Real-World Application

In enterprise environments, RAID levels are selected based on three factors: performance requirements (IOPS and throughput), fault tolerance needs (how many simultaneous drive failures must be survivable), and storage efficiency (what percentage of raw capacity is usable). RAID 5 dominates NAS appliances and mid-tier storage. RAID 10 is standard in production SQL Server and Oracle deployments where downtime is unacceptable.

💡 Pro Tip: On the Tech+ exam, watch for phrases like "maximum speed, no redundancy needed" → RAID 0 · "mirror, 50% capacity" → RAID 1 · "parity, 3+ drives, efficient storage" → RAID 5 · "best performance AND redundancy, 4+ drives" → RAID 10. And always: RAID ≠ backup. This distinction is tested directly.

A technician is diagnosing a desktop that exhibits two separate hardware problems discovered through step-by-step testing. First, she connects a PSU tester and finds the 12V rail reading 11.1V to 13.6V — well outside the ±5% tolerance — explaining the random, load-independent shutdowns the user reported. She replaces the power supply with a known-good unit. The random shutdowns stop, but the system now produces repeating beep codes and fails to complete POST with a blank display. Hint: Focus on “the 12V rail reading 11.1V to 13.6V — well outside the ±5% tolerance.”
🔍 defective power supply + RAM failure — Sequential Testing Reveals Two Distinct Hardware Failures
defective power supply: Voltage rail readings outside ±5% tolerance caused load-independent random shutdowns. Confirmed by PSU tester; eliminated by installing a known-good unit.
RAM failure: After PSU replacement, repeating beep codes and POST failure with a blank display indicate a memory initialization error. One-at-a-time module testing isolates the defective stick — it fails POST in every slot.
Why two failures: An unstable PSU can cause voltage spikes or drops that damage RAM over time. Or both components failed independently. Either way, both must be replaced.
Diagnostic sequence: PSU tester (voltage out of tolerance) → replace PSU → symptoms change but don't fully resolve → beep codes indicate memory error → one-at-a-time RAM test → defective module identified.
Key point: When replacing one component does not fully resolve all symptoms, continue testing — additional hardware failures may be present.

You are the IT systems administrator at Mach Manufacturing Corporation, a CNC machine and ERP operations center. Production line outages cost $50,000 per hour. The plant manager has made it clear: when data is lost, recovery must happen fast — storage cost is secondary. You must configure a Differential Backup strategy that minimizes restore complexity.
// The Organization
Mach Manufacturing Corporation runs CNC controllers and an SAP ERP system from a Windows Server on the plant floor (MACHMFG-ERP01). New production data — schedules, run logs, inventory deltas, quality reports — is generated every business day. Total data: 400 GB. A Sunday Full Backup runs weekly. Daily Differentials Mon–Sat ensure every workday's changes are captured and can be restored in two steps, not five.
// Why Differential Is Right Here
A Differential Backup copies all files changed since the last Full Backup. Wednesday's differential contains Monday's + Tuesday's + Wednesday's changes — all in one set. Restoring requires exactly two backup sets: the last Full + the most recent Differential. For a plant where downtime costs $50K/hour, this two-step restore is the right tradeoff. Storage grows daily but remains manageable. Speed wins over storage efficiency.
// Production Requirements
  • Full Backup every Sunday — complete weekly baseline
  • Differential daily Mon–Sat — cumulative since-Sunday delta
  • Restore must require two sets maximum — no restore chains
  • Recovery time objective (RTO): under 2 hours from any failure
  • Daily coverage of ERP records, CNC logs, and production schedules
// Differential — Key Characteristics
  • Backs up everything changed since the last Full Backup — not since last backup of any type
  • Does NOT clear the archive bit — this is how it accumulates
  • Each day's backup is larger than the previous (Mon < Tue < Wed < Thu)
  • Restore always requires exactly two sets: Full + latest Differential
  • Missing a differential is less risky than incremental — each DIFF is self-contained since the full
