Lesson 1: Hypervisors

Core Principle: A hypervisor is software that allows multiple operating systems to share the same physical hardware simultaneously, managing the allocation of CPU, memory, storage, and network resources to each virtual machine.

📋 Scenario

Your company has five servers that each run one operating system and sit mostly idle. The IT director wants to consolidate them onto a single powerful machine to save money, electricity, and rack space — without losing any of the five OS environments. How can this be done?

How a Hypervisor Works

Think of the hypervisor as a "traffic cop" between physical hardware and the virtual machines that want to use it.

The real machine underneath

Type 1 – Bare-Metal

Runs directly on hardware with no host OS. Used in data centers for maximum performance.

Examples: VMware ESXi, Microsoft Hyper-V (Server), KVM

Type 2 – Hosted

Runs on top of an existing OS. Convenient for desktop virtualization and labs.

Examples: Oracle VirtualBox, VMware Workstation

Key Benefits

Server consolidation, cost savings, easy snapshots & rollback, hardware isolation, testing environments.

Explanation: VMware ESXi runs directly on the hardware with no underlying OS — that is the defining characteristic of a Type 1 bare-metal hypervisor. Type 2 hypervisors (like VirtualBox) run on top of an existing OS such as Windows or Linux.

Virtual Machine Setup Sequence

⬛ DRAG TO REORDER

📋 Scenario

A help desk technician is setting up a test environment on a Type 2 hypervisor (VirtualBox) on their workstation. They have the correct steps but need to put them in the right order. An out-of-sequence procedure could result in a failed installation or misconfigured VM.

Task: Drag the steps below into the correct sequence for setting up a virtual machine using a Type 2 hypervisor. Step 1 should be at the top.
  • ⠿?Allocate resources to the VM: set vCPU count, RAM, and virtual disk size
  • ⠿?Boot the VM and complete the guest OS installation wizard
  • ⠿?Download and install VirtualBox on the host Windows workstation
  • ⠿?Mount the guest OS ISO file (e.g., Ubuntu .iso) to the virtual optical drive
  • ⠿?Verify host machine meets minimum hardware requirements (RAM, storage, CPU virtualization support)
  • ⠿?Create a new VM profile in VirtualBox and select the target OS type

✅ Correct Sequence & Reasoning

  1. Verify host hardware requirements — You must confirm the physical machine has enough RAM, CPU with hardware virtualization (Intel VT-x / AMD-V enabled in BIOS), and storage before installing anything.
  2. Install VirtualBox on the host — The Type 2 hypervisor must be installed on the host OS before any VMs can be created. Without the hypervisor, there is no virtualization layer.
  3. Create a new VM profile — In VirtualBox, you create a VM "container" first and specify the target OS type so VirtualBox can configure default settings appropriately.
  4. Allocate resources — Assign vCPUs, RAM, and virtual disk space to the VM. This must happen before mounting any media or booting.
  5. Mount the ISO file — Attach the guest OS installation media (ISO) to the virtual optical drive so the VM has a source to boot from.
  6. Boot the VM and install — Power on the VM for the first time. It boots from the ISO, and you complete the OS installation wizard just as you would on physical hardware.

❌ Common Sequencing Errors

  • Installing VirtualBox before checking hardware — if the CPU doesn't support hardware virtualization, the install fails or produces errors you can't resolve after the fact.
  • Booting before mounting the ISO — the VM has no boot media and will display an error or attempt to PXE boot (network boot).
  • Skipping resource allocation — VirtualBox won't let you create a VM without assigning at least minimal RAM and disk, but manually skipping or deferring this causes immediate errors.

🌍 Real-World Application

This exact sequence applies whether you're using VirtualBox at home, Hyper-V on a workstation, or provisioning VMs in an enterprise VMware environment. The order of operations — verify hardware, install hypervisor, create VM, allocate resources, attach media, install OS — is universal across all Type 2 hypervisors.

💡 Pro Tip: Always enable hardware virtualization (Intel VT-x or AMD-V) in the BIOS/UEFI before installing a hypervisor. This is the most common reason VM installations fail in lab environments and on the CompTIA exam, this hardware requirement is a tested concept.
