Understanding Tables & Keys

🤔 The Problem

You're building a student enrollment system. How do you uniquely identify each student? What if two students have the same name?

The Challenge of Identification

❌ Without Unique IDs:

NameEmailMajor
James Smithjsmith@university.eduComputer Science
James Smithjsmith2@university.eduBusiness

Problem: Which James Smith are we talking about?

✅ With Primary Keys:

Student_ID PK NameEmailMajor
1001 James Smithjsmith@university.eduComputer Science
1002 James Smithjsmith2@university.eduBusiness

Solution: Each student has a unique identifier!

Primary Key: A column that uniquely identifies each row. Think of it like a Social Security Number — no two people can have the same one.
  • Must be unique — No duplicates allowed
  • Cannot be NULL — Every row must have a value
  • Never changes — Once assigned, it stays the same

🎯 Which Field Should Be the Primary Key?

You're designing a Products table. Which field makes the best primary key?

product_name — e.g. "Wireless Mouse"
product_id — e.g. 12345
price — e.g. 29.99
✅ product_id is ideal: it's always unique, never changes, and is efficient as a number. Product names can duplicate ("Wireless Mouse" from two suppliers) and prices definitely duplicate!

✅ Why Each Answer Is Correct
Create / Import — Uploading a CSV is a bulk data-creation operation. Databases support both manual entry and mass imports from spreadsheets and legacy systems.

Query — Searching for patients by date range and allergy flag is an ad-hoc query: it filters and returns a result set without generating a formatted output document.

Run Reports — A monthly admissions summary is a report: a pre-defined, formatted output aggregating data across many records, typically scheduled for managers or regulators.

Update / Input — Changing an existing patient's address modifies a stored record. The patient already exists; only one field changes — this is distinct from creating a new record.
❌ Common Confusion: Query vs. Report
A query asks an ad-hoc question and returns a raw result set — flexible and real-time. A report is structured, pre-built, and formatted for non-technical stakeholders, often on a schedule. Think: query = searching · report = publishing.
💡 Pro Tip: On Tech+, "generate a summary / produce a printout" → Report · "Find all records where…" → Query · "Add new records / import CSV" → Create · "Modify an existing field" → Update / Input.

Why MySQL? — Choosing the Right Database for the Job

Not all databases are created equal. Before you build one, you need to understand why the choice of database platform matters — and why MySQL is the industry standard for small-to-medium businesses like Mach IT LLC.

🐬 MySQL  — Recommended
✓Free & open-source — no licensing cost for Community Edition
✓MySQL Workbench GUI included — no command-line required
✓Industry standard — powers WordPress, Shopify, Joomla, and millions of SMB apps
✓CompTIA tested — appears on A+ and Server+ certification exams
🪟 Microsoft SQL Server
✗Expensive licensing — Enterprise edition runs $15,000+ per core
✓Tight Azure & Office 365 integration for Microsoft shops
⚠Express (free) edition has 10 GB database size cap
✗Windows-only server (Linux support is newer and limited)
🔴 Oracle Database
✗Enterprise pricing — licensing can exceed $100,000 per processor
✓Unmatched performance for massive transactional workloads
✗Extremely complex to administer — requires dedicated DBAs
✗Overkill for any business under ~500 concurrent users
🐘 PostgreSQL
✓Free & open-source with no restrictions
✓More advanced SQL features than MySQL (JSON, arrays, full-text)
✗No official GUI as polished as MySQL Workbench
⚠Steeper configuration curve — requires more tuning out-of-the-box
✅ Why MySQL is the right choice for Mach IT LLC
  • Zero licensing cost — Community Edition is free forever, unlike SQL Server or Oracle
  • MySQL Workbench provides a professional GUI that mirrors what students will use in the field
  • Scales appropriately for SMBs — handles millions of rows without enterprise overhead
  • Native support in cPanel, GoDaddy, Bluehost, and virtually every web hosting platform
  • The #1 most-deployed open-source RDBMS — the skill transfers to any IT environment
  • CompTIA A+, Server+, and Security+ all reference MySQL concepts in exam objectives

Line-by-Line Breakdown — CREATE TABLE Customers
12 definitions explained
#SQL CodeWhat It Does
1
CREATE TABLE Customers (
Tells MySQL to create a new table called Customers in the active database. The opening parenthesis begins the list of column definitions.
2
customer_id INT AUTO_INCREMENT PRIMARY KEY,
Defines the unique ID for each customer. INT stores whole numbers. AUTO_INCREMENT auto-assigns the next number on every insert. PRIMARY KEY means no two rows can share this value.
3
first_name VARCHAR(50) NOT NULL,
VARCHAR(50) stores variable-length text up to 50 characters. NOT NULL marks this field as required; MySQL will reject any INSERT that leaves this blank.
4
last_name VARCHAR(50) NOT NULL,
Same rules as first_name. Storing names in separate columns allows independent sorting and formatting.
5
company VARCHAR(100),
Optional company name. No NOT NULL — individual customers won't have a company, so NULL is allowed here.
6
email VARCHAR(100) NOT NULL UNIQUE,
Required email address. UNIQUE prevents two customers from sharing the same email and makes email a reliable secondary lookup key.
7
phone VARCHAR(20),
Optional phone number stored as VARCHAR, not INT. Text preserves formatting characters like dashes and parentheses.
8
city VARCHAR(50),
Optional city name. A future query could filter or group customers by city for regional reporting.
9
state CHAR(2),
CHAR(2) stores exactly 2 characters — perfect for US state abbreviations like TX or CA. Unlike VARCHAR, CHAR always reserves the declared length.
10
account_type VARCHAR(20) NOT NULL DEFAULT 'Standard',
Customer tier. NOT NULL ensures it's always set, and DEFAULT 'Standard' means if no value is provided, MySQL automatically assigns Standard.
11
created_date DATE NOT NULL
DATE stores calendar dates in YYYY-MM-DD format. Required (NOT NULL). No trailing comma — this is the last column definition.
12
);
Closes the column definition list. The semicolon terminates the SQL statement and signals MySQL to execute it.
